# Offdex V2 Remote Architecture

## Summary

Build V2 around a managed Offdex remote path, not user-managed VMs and not user-installed Tailscale.

The default connection model is:

- Mac bridge signs into an Offdex account and registers as a trusted machine
- Phone signs into the same Offdex account and sees available machines
- Phone and Mac try a direct encrypted session first
- If direct fails, they automatically fall back to an encrypted Offdex relay
- QR pairing remains for first trust, but remote use no longer depends on a user-managed public relay URL

This replaces “bring your own relay” as the primary architecture. Existing local LAN pairing stays. Existing relay code becomes internal fallback infrastructure, not end-user setup.

## Implementation Changes

### 1. Product and trust model

- Add an Offdex account concept for remote mode only. Local LAN mode still works without an account.
- Keep QR pairing, but change its role:
  - v1 role: embeds bridge URL and optional relay details
  - v2 role: establishes trust between a phone device and a machine device under an Offdex account
- Treat “machine” and “phone” as first-class devices with stable device IDs and long-lived keypairs.
- Persist trusted device identity on both sides. Disconnect removes trust for that phone device only.

### 2. New backend services

Add a minimal Offdex control plane with three responsibilities only:

- `Auth service`
  - email or OAuth-based sign-in for Offdex accounts
  - issues session tokens for phone, web, and bridge
- `Device registry`
  - stores machine records, phone records, pairing relationships, and current reachability state
  - tracks machine online/offline heartbeat
- `Broker/transport service`
  - coordinates remote connection attempts
  - provides direct-session negotiation metadata
  - hosts encrypted relay fallback when direct connection is unavailable

Do not move Codex execution or thread state into the backend. Codex still runs only on the user’s machine.

### 3. Bridge changes

Replace public relay URL configuration with account-backed remote registration.

Bridge responsibilities:

- authenticate with Offdex account
- register the current machine in the device registry
- maintain one outbound secure control connection to Offdex backend
- advertise local capabilities, runtime state, and machine reachability
- accept remote session establishment requests from trusted phone devices
- keep Codex session execution local
- keep LAN `/pairing` and `/pairing.json` for local-first setup

Bridge config changes:

- deprecate `OFFDEX_RELAY_URL` as the primary setup path
- add account/session config for bridge sign-in
- keep existing local host/port config
- keep existing relay internals, but the bridge should treat them as backend-managed fallback transport

### 4. Mobile app changes

Add a real remote-mode user flow:

- sign in to Offdex account
- see trusted machines
- tap a machine to connect
- app silently chooses direct or relay path
- preserve existing local QR/LAN pairing flow for nearby setup

UI changes:

- replace current pairing emphasis on manual bridge URL and manual relay knowledge
- split connection entry into:
  - `Nearby setup`
  - `My machines`
- connection state should surface:
  - local
  - direct remote
  - relayed remote
  - reconnecting
  - offline
- remove remote wording that assumes user-managed relay URLs

State model changes:

- `BridgeWorkspaceState` should evolve into a transport-agnostic machine session state
- add machine identity, account session, and negotiated transport mode
- stop persisting raw relay connection targets in the user-visible flow

### 5. Transport design

Use direct-first, relay-fallback.

Direct path:

- backend brokers connection negotiation only
- bridge and phone exchange ephemeral session metadata
- traffic is end-to-end encrypted between phone and bridge
- if direct path succeeds, use it for live snapshots, turn submission, interrupts, and health

Relay fallback:

- reuse the existing relay concepts, but move them behind the Offdex backend
- relay only forwards opaque encrypted frames
- backend cannot read Codex content
- fallback is automatic and invisible to the user

Transport rules:

- same high-level APIs for local, direct remote, and relayed remote
- transport selection happens below the UI state layer
- reconnect policy prefers the last known good path, then falls back automatically

### 6. Public interfaces and type changes

Add these concepts to shared protocol/types:

- `OffdexAccountSession`
- `OffdexDeviceId`
- `OffdexMachineRecord`
- `OffdexTrustedDeviceRecord`
- `OffdexTransportMode = "local" | "direct" | "relay"`
- `OffdexRemoteCapability`
- `OffdexConnectionTicket` or equivalent broker-issued session descriptor

Deprecate current user-facing dependence on:

- pairing payload relay URL
- pairing payload room ID
- pairing payload relay secret as a primary remote UX surface

Keep existing pairing payload backward-compatible for local mode during migration.

### 7. Security model

Use device and session keys, not user-managed networking.

Required properties:

- each phone and machine has a stable device keypair
- QR pairing establishes device trust
- backend authenticates devices and routes connection setup
- all thread/control traffic is encrypted end-to-end between phone and bridge
- relay fallback carries ciphertext only
- disconnect revokes that phone device’s trust for that machine
- machine offline state must not leak Codex content, only presence metadata

### 8. Rollout shape

V2 should ship in this order:

1. account and device registry
2. bridge sign-in and machine registration
3. mobile sign-in and machine list
4. local QR pairing upgraded to trusted device pairing
5. backend-brokered relay fallback using existing relay logic
6. direct path optimization
7. final UI polish and copy cleanup

Important constraint:
- do not ship Tailscale mode or self-host mode in V2
- do not expose user networking choices in the main UI
- keep advanced networking modes for a later power-user phase

## Test Plan

### Core behavior

- sign in on bridge and mobile under same account
- pair phone to machine once via QR
- machine appears in mobile machine list
- phone reconnects to same machine after app restart
- phone reconnects after background/foreground
- machine offline appears correctly without stale “connected” UI

### Transport behavior

- local LAN connection still works with no account remote session active
- remote direct connection succeeds when both endpoints are reachable
- relay fallback takes over when direct connection fails
- reconnect resumes on the same machine after temporary network loss
- turn send, interrupt, runtime switch, and live snapshot updates work identically across local/direct/relay

### Security behavior

- untrusted phone cannot connect to machine
- revoked phone trust loses access immediately
- relay cannot read workspace payloads
- wrong connection ticket / expired broker ticket is rejected
- machine-to-phone trust survives app restart but is removed on disconnect

### Migration and compatibility

- old local pairing still works during transition
- existing bridge-only users can continue local use without account sign-in
- old relay-specific persisted connection targets are ignored or migrated cleanly

## Assumptions and defaults

- Remote mode requires an Offdex account.
- Local LAN mode remains available without account dependency.
- Direct-first with encrypted relay fallback is the default and only V2 remote UX.
- Self-hosted relay mode and Tailscale-style power-user mode are out of scope for V2.
- Existing relay code is retained as implementation substrate, but no longer treated as user-managed infrastructure.
- Backend scope is limited to auth, device registry, brokering, and encrypted relay transport. Codex execution stays local on the machine.
