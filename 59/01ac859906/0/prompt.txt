# Offdex Final Connectivity Refactor

## Summary

Build the final user-facing connectivity model around machines, not URLs:

```text
Pair once -> choose machine -> use Codex anywhere
```

Internally, Offdex will support exactly two transports:

```text
local
relay
```

`direct remote` will be removed from the main path because true NAT traversal is not currently worth the reliability and complexity cost. The product will prefer local when reachable and automatically fall back to Cloudflare relay when the user is away.

## Architecture

- Keep Vercel for:
  - landing page
  - WebUI at `/app`

- Add Cloudflare Workers + Durable Objects for:
  - machine presence
  - QR pairing claim
  - trusted device sessions
  - relay WebSocket rooms
  - encrypted proxy/live frame forwarding

- Keep Codex local:
  - Codex still runs only through `codex app-server` on the user’s machine.
  - Offdex never stores ChatGPT credentials.
  - The bridge remains the source of truth for threads, messages, and runtime state.

## Product Flow

- User installs CLI:
```bash
npm install -g @dhruv2mars/offdex
```

- User runs:
```bash
offdex start
```

- Terminal shows:
  - machine name
  - connection status
  - QR
  - WebUI link

- Phone or WebUI pairs once.
- Client stores a trusted machine session.
- After that, the user sees:
```text
Dhruv’s MacBook
Online
Connect
```

- Client behavior:
```text
try local bridge
if local fails, use Cloudflare relay
```

- UI should only expose:
```text
Connected locally
Connected remotely
Offline
```

No relay URL, control-plane URL, ticket, room, or bridge URL in the normal UI.

## Implementation

- Add `packages/control-plane-worker` with Wrangler config and a Durable Object relay/control-plane.
- Reuse the current control-plane API shape where practical, but make it Cloudflare-compatible:
  - no Bun server
  - no filesystem state
  - Durable Object owns room/session state
  - Worker routes HTTP and WebSocket traffic to the relevant Durable Object
- Refactor shared protocol:
  - change `OffdexTransportMode` to `"local" | "relay"`
  - add machine/session naming around `machineId`
  - keep old direct/relay URI decoding only for migration compatibility
- Refactor mobile client:
  - introduce `MachineSession` / `TransportManager`
  - remove direct-remote as a user-facing state
  - persist trusted machine session
  - connect local first, relay fallback second
- Refactor web `/app`:
  - use the same machine-session mental model
  - support terminal-provided pairing/link flow
  - stop behaving like a bridge URL form first
- Refactor bridge:
  - register machine with Cloudflare control-plane when configured/defaulted
  - maintain host WebSocket to relay
  - keep local bridge available
  - keep startup output simple
- Refactor CLI:
  - default to Cloudflare control-plane URL after deployment
  - keep advanced override via `OFFDEX_CONTROL_PLANE_URL`
  - status should show local/remote readiness without exposing internals by default

## Deployment

- Use Wrangler login locally for first deployment.
- Deploy to Cloudflare Workers free tier using generated `workers.dev` URL.
- After deploy, set that URL as the default managed remote endpoint in the CLI/bridge.
- Add CI workflow for Cloudflare deploy only when control-plane worker files change.
- Keep Android/mobile release opt-in and separate from CLI/web/relay changes.

## Tests

- Unit:
  - local transport succeeds
  - local failure falls back to relay
  - relay proxy round-trips encrypted payloads
  - untrusted sessions are rejected
  - revoked sessions cannot connect
  - old pairing links still decode

- Integration:
  - bridge registers machine
  - QR claim creates trusted session
  - client lists trusted machine
  - client connects locally when reachable
  - client connects remotely through relay when local is unreachable
  - turn send and live snapshot work over both transports

- Manual:
  - `offdex start`
  - pair mobile
  - pair web `/app`
  - send Codex turn locally
  - force local unreachable
  - reconnect through relay
  - verify Mac online + internet is enough for remote usage

## Assumptions

- Cloudflare Workers/Durable Objects free `workers.dev` URL is acceptable.
- Wrangler login is the deployment auth path.
- No Offdex account in this phase.
- No Tailscale.
- No VM.
- No real direct NAT traversal until a later optimization.
