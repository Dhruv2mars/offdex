# Product

## Register

product

## Users

Primary users are daily Codex users who already trust Codex for real coding work and want the same workflow away from their desk. They move between desktop, web, and phone, often while a live Codex session keeps running on their Mac. They need to monitor, steer, approve, review, and continue real work without learning a second product.

Secondary users are new Offdex users installing through the CLI who need pairing, reconnect, and account state to feel obvious and safe from the first session.

## Product Purpose

Offdex exists to make Codex usable from anywhere without turning Codex into a separate cloud product or a watered-down companion app. The bridge on the user's machine owns authentication, workspace access, tools, and runtime truth. Web and mobile clients are fast, polished windows into that same live session.

Success means a normal Codex user can open Offdex WebUI and do the same browser-safe work they would do in Codex Desktop: start or resume threads, watch streaming work, inspect diffs and command output, approve actions, manage settings, use plugins and connectors, and recover cleanly after disconnects.

## Brand Personality

Exacting, calm, live.

Offdex should feel technically serious, immediate, and trustworthy. It should never feel like a generic chatbot, a toy remote control, or a dashboard that hides runtime truth behind marketing gloss.

## Anti-references

- Generic AI chat apps that flatten everything into bubbles and hide the actual work.
- Companion-only mobile or web clients that can supervise but not really operate Codex.
- Loud SaaS dashboards with decorative gradients, inflated metrics, and invented controls.
- Sync-heavy products that feel disconnected from the real machine state.

## Design Principles

- Bridge owns truth. Auth, tools, workspace access, permissions, and recovery come from the machine runtime, not the browser client.
- Desktop parity where browser allows. If Codex Desktop supports a browser-safe workflow, Offdex WebUI should support it too.
- Show work, not summaries. Commands, diffs, plans, approvals, connector state, and errors should stay visible and inspectable.
- Direct over sprawling. Fast access to the active thread and right inspector matters more than deep settings mazes.
- Remote confidence matters. Users should always understand whether they are connected, what machine they are using, what Codex is doing, and what needs their attention.

## Accessibility & Inclusion

Ship at WCAG AA contrast or better. Every major action must be keyboard reachable. Focus states must stay obvious. Reduced-motion users should not depend on animation for comprehension. Dense workbench layouts must still preserve readable type, clear labels, and sufficient target sizes on desktop and mobile-class browsers.
